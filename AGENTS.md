# Agent Guide

This file is the quick-start context for future agents working in this
repository. Prefer it for orientation, then follow the deeper docs linked below
when changing a specific subsystem.

## Command Output Policy

`@RTK.md` enables the `rtk` command text limiter. Prefix shell commands with
`rtk` so large command output is truncated before it consumes the conversation
context.

Examples:

```bash
rtk git status
rtk rg -n "controller_db_url|auth_db_url" .
rtk uv run pytest tests/
```

If you need raw, unfiltered output for a small command, use `rtk proxy <cmd>`.

## Common Commands

Run the full Python test suite with:

```bash
rtk uv run pytest tests/
```

Focused Python tests:

```bash
rtk uv run pytest tests/test_api.py -v
rtk uv run pytest tests/test_persistence_db_infra.py tests/test_alembic_config.py -v
rtk uv run pytest tests/test_models_backfill.py -v
```

Frontend install, test, and build:

```bash
cd frontend
rtk npm ci
rtk npm test
rtk npm run build
```

Local backend and full-stack helpers:

```bash
rtk scripts/start_controller.sh
rtk scripts/start_agent.sh
rtk scripts/start_controller_stack.sh
rtk scripts/start_agent_stack.sh
rtk scripts/dev_fullstack.sh
rtk python scripts/backfill_models_db.py --config config.yaml
```

## Runtime And Generated Files

- `.llama_pack.env` is generated by onboarding scripts and contains local
  secrets. Do not commit or print it unless the user explicitly asks for a
  specific value.
- `config.yaml` is local runtime config. Prefer `config.example.yaml` and the
  docs when explaining defaults.
- `logs/` contains runtime logs, SQLite databases, downloads, conversion output,
  quantization output, and other local state.
- `llama_pack/ui/react/` is generated by `cd frontend && npm run build` and
  is served by FastAPI for the production UI.
- `frontend/node_modules/`, `.venv/`, `.pytest_cache/`, `tests/tmp/`, and other
  cache/build folders are local artifacts. Avoid editing or committing them.
- This repo can have unrelated in-progress user changes. Check `rtk git status`
  before editing and do not revert changes you did not make.

## Architecture Map

Runtime modes are controlled by `AppConfig.mode`:

- `agent`: runs on a model host and manages local `llama-server` processes.
- `controller`: runs centrally, tracks nodes, proxies operations, and manages
  durable orchestration.

Main ownership boundaries:

- `llama_pack/main.py`: FastAPI app creation and app-state wiring.
- `llama_pack/api/`: HTTP request/response translation, dependency wiring,
  validation, and error mapping.
- `llama_pack/core/`: domain logic for chat routing, node management,
  model assets, orchestration, persistence, runtime process management, tools,
  memory, benchmarks, and threads.
- `llama_pack/providers/` and `llama_pack/storage/`: external command and
  storage/provider primitives.
- `frontend/src/api/`: typed frontend API helpers by backend domain.
- `frontend/src/pages/`, `frontend/src/components/`, and
  `frontend/src/features/`: React UI surfaces and pure UI/domain helpers.

Keep route handlers thin. Business rules, retries, state transitions, and
payload shaping that is not purely HTTP-specific usually belong in `core`.

When the same behavior is needed in multiple places, prefer extracting only as
much abstraction as the reuse justifies. Small pure functions should usually
live in a nearby helper module such as `frontend/src/features/...` or the
relevant `core` package. Larger behavior with state, dependencies, or a clear
domain role should move into its own class or service instead of being copied
between pages, routes, or managers.

## Databases And Migrations

Persistence is SQLAlchemy-backed and Alembic-managed. The default local backend
is SQLite under `logs/`, but database URLs can be overridden in config.

Configured database URLs:

| Config key | Default SQLite file | Alembic target | Purpose |
| --- | --- | --- | --- |
| `controller_db_url` | `logs/controller_state.db` | `controller` | orchestration, durable queue, attempts, events, controller state |
| `auth_db_url` | `logs/auth_store.db` | `auth` | admin keys, agent keys, external key metadata |
| `audit_db_url` | `logs/audit_events.db` | `audit` | audit/event records |
| `chat_sessions_db_url` | `logs/chat_sessions.db` | `chat_sessions` | durable chat sessions |
| `downloads_db_url` | `logs/downloads.db` | `downloads` | model download records |
| `benchmarks_db_url` | `logs/benchmarks.db` | `benchmarks` | benchmark definitions, runs, and samples |
| `models_db_url` | `logs/models.db` | `models` | model catalog, assets, profiles, and deployments |
| `settings_db_url` | `logs/settings.db` | `settings` | durable application settings |

For normal maintenance, run every migration target with the single project
command:

```bash
rtk uv run python scripts/migrate_all.py --config config.yaml
```

Migration branches live in `migrations/versions/<target>/`. When debugging or
creating a target-specific migration manually, always target the branch
explicitly.

If `-x db=` is omitted, Alembic defaults to `controller`. Avoid unqualified
`head` because this project has multiple migration branches.

For fresh controller setup, prefer:

```bash
rtk scripts/onboard_controller.sh
```

That script writes `.llama_pack.env`, runs migrations, creates the first
admin API key, and prints registration material for agents.

## Testing Notes

- Use `rtk uv run pytest tests/` for the full suite.
- Many persistence tests create temporary SQLite files with helpers in
  `tests/persistence_db_setup.py`.
- Tests should not create keys or mutate real repo databases under `logs/`.
- Frontend tests run through Vitest with `cd frontend && rtk npm test`; Python
  wrappers also cover frontend/static-serving behavior in
  `tests/test_frontend_tests.py`, `tests/test_ui_static_serving.py`, and
  `tests/test_package_data.py`.
- When changing migrations or persistence models, add or update focused tests
  around the relevant store and Alembic target.
